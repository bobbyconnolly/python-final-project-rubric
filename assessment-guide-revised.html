Project Requirements

Core Architecture

Your project must implement one of the following architectures:

Option 1: Web Application (Recommended)

  • Python backend using Flask or Django
  • SQLite database integration
  • Web-based frontend

Option 2: Standalone Application

Your project could be a game, visualization tool, data processor, or creative application - for example, making a Pygame window that renders graphics, creating generative art, applying image effects, analyzing data sets, or calling external APIs.

Technical Requirements & Assessment Criteria

Core Development (1.1, 2.2, 2.3 - totalling 14 marks)

These criteria evaluate how well your project is constructed overall:

  • Does it work as a complete, polished product?
  • Is the code well-organized and maintainable?
  • Would another developer understand how your project is structured?

3.1 Debugger Usage (3 marks)

During your assessment meeting, you must demonstrate:

  • Setting and using breakpoints in PyCharm
  • Inspecting variables during runtime execution
  • Step-through debugging of a meaningful interaction (e.g., route handling, data processing)

3.3 Debugging Strategies (3 marks)

You must include at least two (2) print statements in any of the following key areas of your program:

  • Before and after database operations (e.g., print(f"Inserting record: {data}"))
  • When routes/endpoints are hit (e.g., print(f"POST /api/users with data: {request.json}"))
  • When key functions are called with their parameters (e.g., print(f"calculate_total called with items: {items}"))
  • Before returning results (e.g., print(f"Returning result: {result}"))
  • When errors occur (e.g., print(f"Error in process_data: {str(e)}"))

3.5 Solution Documentation (2 marks)

Create a SOLUTIONS.md file in your project repository documenting one significant issue you encountered. Use this template:

# Issue: [Brief Title] ## Problem Description - What was the unexpected behavior? - What should have happened instead? - How was this issue discovered? ## Root Cause Analysis - What was the underlying cause? - What assumptions were incorrect? - What dependencies were involved? ## Resolution - How was it fixed? (or planned fix if unresolved) - What changes were made? - What alternatives were considered? ## Prevention - How can similar issues be prevented? - What lessons were learned? - What warning signs should be watched for?

2.1 Backup Strategy (2 marks)

Your project must be uploaded to GitHub, preferably in a subfolder of your existing python-final-project-2024 repository. If you cannot add it to this repository, creating a new repository is acceptable.

Need help with GitHub? Don't hesitate to reach out - I'm here to help!

2.2 Access Rights (2 marks)

You only need to complete ONE of the following options:

a. Add your professor or a fellow student as a GitHub collaborator with read access
(Recommended: This is the easiest way to get these 2 marks!)

b. Introduce a Python class into your program that demonstrates access control using private attributes with underscore prefix ("_")

c. Implement a custom access control feature in your application

2.3 Security - Firewall (2 marks)

For web applications:

  • Demonstrate port blocking/unblocking using your system's firewall
  • Show how it prevents client-server communication when the port is blocked
  • See the Firewall-Port-Blocking-Guide.pdf guide on Omnivox for how to do this

Note: Projects without a webserver must still demonstrate this during the meeting using a simple client-server project in PyCharm. This demonstration code doesn't need to be on GitHub.

2.4 Security - Encryption (2 marks)

Implement a feature that stores sensitive information in your SQLite database in an encrypted format:

  • Store encrypted personal information in the database
  • Use an environment variable to store the encryption key
  • Demonstrate encryption/decryption during runtime
  • See the Database-Encryption-Guide.pdf guide in the Final Project section on Omnivox documents

Note: Projects without a database must still demonstrate this during the meeting using a simple project in PyCharm. This demonstration code doesn't need to be on GitHub.

  • Code is pushed to GitHub
